## 「同じものを頼んだのに、なぜこんなに違うのか」

システム開発で3社相見積もりを取ると、こんな結果になることが普通にあります。

- A社：800万円
- B社：2,400万円
- C社：4,000万円

同じRFPを渡したはずなのに、なぜここまで違うのか。
答えは、**RFPに書かれていない前提を、各社が自分の判断で埋めているから** です。

このコラムでは、見積もりがブレる5つの原因と、**比較可能な相見積もりを取るための実践方法** を解説します。

---

## 見積もりがブレる5つの原因

### 原因1：スコープ解釈の幅

「ECサイトを作りたい」と書かれていても、

- A社：「商品ページとカートだけ」と解釈
- B社：「決済・在庫・会員管理込み」と解釈
- C社：「マルチテナント・多言語・運用画面まで」と解釈

スコープの揺れだけで、見積もりは3倍以上変わります。

### 原因2：非機能要件の前提

「ユーザー数」「同時アクセス数」「可用性」「セキュリティ要件」が明記されていないと、各社は **自社の標準** で勝手に前提を置きます。

- 月間100ユーザー想定 → AWS Free Tierで足りる
- 月間100万ユーザー想定 → 冗長構成・CDN必須

この前提差が数倍の差になります。

### 原因3：技術スタックの自由度

「技術選定はお任せ」と書いてあると、

- A社：自社が得意なPHP + 自前FW（安いが拡張性に難）
- B社：Next.js + Supabase（モダンだが運用知見が要る）
- C社：エンタープライズグレードのJava構成（堅牢だが高額）

どれが正解とは言えませんが、**比較不可能な見積もり** が生成されます。

### 原因4：開発体制の違い

- 1人のフルスタックエンジニアで回すA社
- フロント／バック／インフラの分業体制のB社
- PM＋技術リード＋実装者の階層型のC社

同じ機能を作っても、体制によって人月単価×期間が大きく変わります。

### 原因5：リスクバッファの取り方

経験豊富なベンダーほど、要件の不確実性に対して **リスクバッファ** を上乗せします。

- A社：バッファ20%
- B社：バッファ50%
- C社：バッファ100%（請負で確実に黒字化したい）

不確実性が高いほど、バッファ差が金額差を生みます。

## 比較できる見積もりを取るための5ステップ

### ステップ1：スコープを「機能名」ではなく「ユーザーストーリー」で書く

「商品検索機能」ではなく、

```
As a 一般ユーザー
I want to カテゴリと価格帯で商品を絞り込みたい
So that 候補を比較しやすくする
```

ユーザーストーリーで書くと、**そのストーリーに含まれる詳細仕様の解釈幅が狭まり**、各社の理解が揃います。

### ステップ2：非機能要件の数値を明記する

最低限以下を書きます。

| 項目 | 記入例 |
| --- | --- |
| 想定ユーザー数 | DAU 500、ピーク時同時 50 |
| 想定データ量 | 商品10万件／注文1,000件/日 |
| レスポンス目標 | 主要画面 2秒以内 |
| 可用性 | 月間稼働率99%（夜間メンテ可） |
| セキュリティ | SSL／個人情報あり／PCI DSS対象外 |

これだけで、見積もりのブレ幅は半減します。

### ステップ3：技術スタックの「制約」と「自由度」を分ける

- **制約**：AWS必須／Python/Node.jsから選定／既存DBはMySQL継続
- **自由**：FW・テストツール・デプロイツールはベンダー判断

「全部お任せ」だと比較不可能、「全部指定」だとベンダーの強みが消えます。**制約2〜3項目、残りは自由** が現実的なバランスです。

### ステップ4：見積もりの「内訳」を必須項目にする

提案フォーマットで以下を必須にします。

- フェーズ別の人月（要件定義／設計／実装／テスト／リリース）
- 役割別の単価（PM／リード／実装／QA）
- リスクバッファの内訳と理由
- 追加要件発生時の単価

総額だけ提示する見積もりは比較不可能なので、断った方が安全です。

### ステップ5：質疑応答セッションを設定する

提案を受け取って終わりではなく、各社1時間ずつ **質疑応答** を設定します。

確認すべきは、

- 想定リスクのトップ3と、その対処方針
- 似たプロジェクトの実例（規模・期間・体制）
- 進め方のイメージ（ウォーターフォール／アジャイル／ハイブリッド）

紙の提案より、質疑応答での回答の解像度が、最終判断の決め手になります。

## 「安い見積もり」を選ぶときのチェックポイント

A社（800万円）が魅力的に見えても、すぐに飛びつくのは危険です。最低限以下を確認します。

- スコープが本当に同じか（機能の数・非機能要件）
- リスクバッファが極端に薄くないか（追加請求の温床）
- 体制が薄すぎないか（PMなし／テスト工数なし）
- 過去実績の規模感が同等か

「不当に安い」場合、**見積もり時点でのスコープ漏れ** が後の追加請求として返ってきます。詳しくは [パートナー選定の失敗事例](/column/partner-failure-cases) も参照してください。

---

## まとめ：比較可能な見積もりは、発注側のRFPで決まる

見積もりが3〜5倍違うのは、ベンダーが悪いのではなく **RFPに書かれていない前提が多すぎる** ことが大半の原因です。

- スコープはユーザーストーリーで書く
- 非機能要件は数値で書く
- 技術制約は2〜3項目に絞る
- 見積もり内訳を必須項目にする
- 質疑応答セッションを設定する

この5ステップを踏めば、見積もりは「比較可能な意思決定材料」になります。
