なぜ「APIを叩くだけ」では業務システムにならないのか

「ChatGPT API は数行で叩けます」「Claude API も同じです」――これは事実です。fetch 1回で動くサンプルコードは公式ドキュメントにいくらでもあります。

ですが、それは PoC レベル の話です。業務システムとして本番運用するには、API 呼び出しの周辺で次のような問題が必ず発生します。

  • ユーザーが20秒待たされる（LLMの応答が遅い）
  • 同じ質問に毎回違う答えが返る（出力が安定しない）
  • JSON で返してくれと指示したのに自由記述で返ってくる
  • API 利用料が想定の3倍になっていた
  • インシデント時にどの会話が原因か追えない
  • LLM プロバイダの障害でシステム全体が停止する

これらは技術ブログのサンプルでは出てきませんが、業務利用では必ず直面します。本記事では、ChatGPT／Claude API を業務システムに組み込むときに「最初から考えておくべき設計ポイント」を整理します。

1. 構造化出力を前提に設計する

業務システムでは、LLM の出力を 後続処理にそのまま使える形式 で受け取る必要があります。「自由記述で返ってきた回答を正規表現でパース」する設計は、本番で必ず破綻します。

推奨アプローチ

Function Calling / 構造化出力を使う
  • OpenAI: response_format: { type: "json_schema", schema: {...} }
  • Anthropic: tool_use による構造化レスポンス
  • JSON Schema をAPI側に渡し、必須フィールド・型・列挙値を強制する
  • パース失敗時のフォールバック処理を必ず実装（再試行 or 人手介入キュー）

「プロンプトで JSON で返してと指示する」だけでは、稀に別形式で返ってきたり、JSON が壊れていたりします。Schema 強制を使えるなら必ず使います。

2. タイムアウトとリトライ戦略

LLM API は、通常の REST API より遥かに応答時間が長く、不安定です。

シナリオ 応答時間目安
短文の分類・抽出 1〜3秒
通常の Q&A 応答 3〜8秒
長文要約・複雑推論 10〜30秒
ストリーミング全体完了まで 30〜60秒

設計上のポイント

  • 同期処理しない: HTTP リクエスト処理の中でLLM API を待つ設計は、ユーザー体感が悪く、サーバーリソースも食う
  • ストリーミングを使う: ユーザー向けチャットなら必ず Server-Sent Events で逐次表示
  • バックグラウンドジョブ化: 要約・分類などはキューに入れて非同期処理
  • タイムアウト: クライアント側 60秒、サーバー側 120秒程度が目安
  • リトライ: 一時的エラー（429 / 5xx）は指数バックオフで最大3回、ただし冪等性に注意

3. 冪等性とコスト爆発の防止

LLM API は 1リクエスト数円〜数十円 かかります。リトライやバグで二重実行されると、月末の請求で気づくパターンが発生します。

必須の防御策

コスト爆発の典型パターンと対策
  • 無限リトライ: ジョブが失敗→自動リトライが永遠に続く。最大試行回数を必ず設定
  • 冪等キー: 同じ入力に対する2重実行を防ぐ。リクエストIDをDB保存し、結果をキャッシュ
  • レート制限: ユーザーごと、テナントごと、組織全体の3層で制限値を持つ
  • 月額アラート: API ダッシュボードでコスト閾値を設定、Slack通知
  • テスト環境のキー分離: 開発用と本番用のAPIキーを完全に分け、開発環境は安いモデル＋低レート制限

特に テスト環境で本番キーを使ってループ処理を回した結果、月数十万円の請求は実際によくあります。最初から防御層を入れておくべきです。

4. プロンプトのバージョン管理

業務システムでは、プロンプトはコードと同等に重要なロジックです。

よくある失敗パターン

  • プロンプトをハードコードして git に直書き → 変更履歴が追えない、A/Bテストできない
  • 管理画面でプロンプトを編集できるようにした → 誰がいつ変えたか追えなくなり、本番で壊れる
  • プロンプトを変えたら全機能の精度が変わった → 影響範囲が読めず怖くて変えられなくなる

推奨アプローチ

  • プロンプトはバージョン管理されたファイル（YAML / JSON）で管理
  • 各プロンプトに version, owner, last_evaluated_at をメタデータとして保持
  • プロンプト変更時は、評価データセットで自動回帰テスト
  • 本番デプロイ前に必ず精度ベンチマークの結果を比較

LangSmith / Langfuse / 自前評価基盤いずれかで、プロンプトとその精度結果をペアで管理する仕組みを最初から入れる方が後で楽です。

5. ハルシネーション対策と根拠提示

LLM はもっともらしい嘘を返すことがあります（ハルシネーション）。業務システムでは、この性質を 設計でカバーする 必要があります。

対策パターン

業務利用でのハルシネーション対策
  • 出典を必ず出力させる: RAGなら検索ヒットした文書ID、社内DBならレコードIDを回答に紐付ける
  • 不確実性の伝達: 「資料に該当情報なし」と答えさせる。プロンプトで「分からないときは推測せず分からないと答えよ」を明示
  • 検索ヒット数による分岐: RAGで参照文書がゼロ件なら、LLMに渡さず「該当なし」を即返す
  • 金額・日付・固有名詞の抽出は二段階: LLMに「該当箇所」を抽出させ、正規表現で値を取り出す
  • 業務影響の大きい判断は人間レビュー: 「LLMが下書き、人間が承認」のフロー設計

「ハルシネーションをゼロにする」は不可能なので、ゼロを目指すのではなく、業務影響を許容範囲に抑える設計が現実解です。

6. 監査ログとオブザーバビリティ

業務利用では、誰が・いつ・何を質問し、LLMが何を回答したか、を全件保存することが基本です。

最低限保存すべきデータ

項目 用途
ユーザーID／部門 利用統計、権限制御
タイムスタンプ 監査ログ
プロンプト全文（システム＋ユーザー） 再現・調査
モデル名／バージョン 精度劣化時の原因調査
トークン数（入力／出力） コスト分析
RAG参照ドキュメントID 出典確認、機密情報漏洩調査
ユーザーフィードバック（👍/👎） 精度改善の手がかり
応答時間 パフォーマンス監視

これらを後から追加するのは大変なので、最初から保存する設計にします。データ量は意外と多いので、ホットストレージ（最近30日）と コールドストレージ（90日以降）で分けると運用コストを抑えられます。

7. プロバイダロックインへの備え

ChatGPT／Claude／Gemini のいずれも、API仕様変更とモデル廃止が日常的に発生します。

  • OpenAI: 旧モデル（gpt-3.5系）は段階的に廃止
  • Anthropic: Claude 2 → 3 → 3.5 → 4 → 4.5 → 4.6 → 4.7 と継続的に世代交代
  • Google: Gemini も同様

業務利用なら、最初から「モデルを差し替えられる構造」で実装します。

推奨アーキテクチャ

  • LLM 呼び出しを抽象化レイヤで包む（プロバイダ抽象化）
  • プロンプトとモデルパラメータをコードから分離（YAML / DB管理）
  • LiteLLM / Vercel AI SDK / 自前ラッパー、いずれかでプロバイダ切替可能に
  • 精度評価データセットを保持し、新モデルで自動評価できる環境を持つ

「Claude Sonnet 4.6 で精度が出たので、4.7 に上げてみたら回帰した」が頻繁に起こるので、評価ループなしの本番運用は事故ります。

まとめ: 業務システムに必要な「LLM周辺装置」

ChatGPT／Claude API を業務システムに組み込むとき、API 呼び出しそのものはコード全体の1割程度です。残り9割は次の「周辺装置」になります。

  1. 構造化出力とパース失敗対策
  2. タイムアウト・リトライ・非同期化
  3. コスト制御（冪等キー・レート制限・アラート）
  4. プロンプトのバージョン管理と回帰テスト
  5. ハルシネーション対策（出典提示・不確実性伝達）
  6. 監査ログとオブザーバビリティ
  7. プロバイダ抽象化と移行容易性

「PoC が動いた → 本番化はすぐできる」と思っていると、ここで大きく工数が膨らみます。最初から本番化を見据えた設計にすると、トータルの開発期間は短くなります。
