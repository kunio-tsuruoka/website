「生成AI開発、いくらかかるんですか？」に答えづらい理由

「ChatGPTを業務に組み込みたい」「社内データでRAGを作りたい」――生成AIの相談を受けて最初に問われるのは、ほぼ100%この質問です。

しかし、率直に言ってこの質問にはそのままでは答えられません。なぜなら、生成AI開発は「何を作るか」よりも「どこまで作るか」で費用が大きく変わるからです。

具体的には次の3軸で予算規模がまったく違います。

生成AI開発の費用を決める3軸
  • フェーズ: PoC（実現可能性検証）／プロトタイプ／本番運用、どこまでやるか
  • データ統合の深さ: 単発のプロンプト呼び出しか、社内データを参照するRAGか、業務システム連携まで含めるか
  • 運用要件: 内部利用か、顧客向けか、SLA・監査ログ・セキュリティ要件の有無

本記事では、この3軸を踏まえて生成AI受託開発の費用相場をフェーズ別に分解し、「見積書のどこを読めばよいか」発注者目線で解説します。

フェーズ別の費用相場

中堅企業の業務システムに生成AIを組み込むケースを想定した、ざっくりした目安です。実際の見積もりは要件次第で前後します。

フェーズ 期間目安 費用レンジ 主な成果物
PoC（実現可能性検証） 2〜4週間 80万〜250万円 検証レポート、サンプルプロンプト、精度評価
プロトタイプ開発 1〜2か月 200万〜600万円 動くデモ、限定ユーザー試用版
本番化（MVP） 2〜4か月 500万〜1,500万円 本番運用可能なシステム、認証・ログ・監視一式
機能拡張・運用 継続 月20万〜100万円 運用保守、追加機能、精度改善

※ 上記は社内向け業務支援AI（チャット、要約、分類、ドキュメント検索など）の場合。顧客向けプロダクトや音声・画像を扱う場合はさらに拡大します。

内訳: 生成AI開発で「実際に何にお金がかかるか」

1. 要件定義・PoCで見落とされやすい費用

PoCで一番削られがちなのが、評価設計の工数です。「LLMに投げて、それっぽい回答が返ってきた」では本番化はできません。

PoCで本来必要なのは次の作業です。

  • 評価データセットの作成（業務担当者の協力が必須、30〜100件程度）
  • 評価指標の定義（精度・網羅性・応答時間・ハルシネーション率）
  • 複数モデル／複数プロンプトの比較検証
  • 失敗パターンの分類と再現性確認

これを軽視したPoCは「動いた／動かない」しか分からず、本番化判断ができないままお金だけ消えます。

2. プロトタイプ・本番化で大きい3つの費用

① RAG基盤構築（社内データ参照型の場合）

社内ドキュメント・FAQ・データベースをLLMに参照させる場合、ベクトルDB構築・チャンク分割設計・検索チューニングで開発工数の30〜50%を占めます。「データを入れれば動く」ものではありません。

② 認証・権限・監査ログ

業務利用するなら、誰がいつ何を質問したかのログ、部門別の参照権限、機密情報のマスキングが必須。これだけで100〜300万円規模の追加工数になることがあります。

③ プロンプトエンジニアリング・精度改善

「動く」と「使える」の差は、プロンプト設計と評価ループに集約されます。本番化フェーズの20〜30%は精度改善に使われるのが標準です。

3. ランニングコスト: LLM API利用料の現実

開発費用と別に、運用開始後のLLM API利用料が発生します。よくある誤解は「Claude API は無料か激安」というものですが、業務利用では月数万〜数十万円になります。

目安（2026年時点、為替レート150円/USDで概算）：

  • Claude Sonnet で社内100名・1人1日10回利用 → 月3万〜8万円
  • GPT-4o で社内500名・1人1日20回利用 → 月15万〜40万円
  • 大量バッチ処理（要約・分類）月100万件 → 月20万〜80万円

RAGの場合は埋め込み（Embedding）コストとベクトルDBホスティング費（月1万〜10万円）も追加されます。

見積書チェックリスト: 発注者が確認すべき10項目

必ず質問すべき項目
  1. PoCの評価指標と評価データセット作成は工数に含まれているか
  2. 使用予定のLLMモデル名とバージョン（Claude Sonnet 4.x／GPT-4o など具体名）
  3. LLM API利用料は誰が負担するか（受託側／発注側）、月額の試算は出ているか
  4. RAGを使う場合、ベクトルDBの選定理由（Pinecone／pgvector／Azure AI Search 等）
  5. 機密データの取り扱い（学習に使われない設定、データ保持期間、リージョン）
  6. 監査ログと利用状況ダッシュボードは含まれているか
  7. ハルシネーション対策（出典表示、根拠抽出、不確実性の伝達）
  8. 運用保守費用の月額と、含まれる作業範囲（精度改善、モデル更新対応、障害対応）
  9. API仕様変更・モデル廃止時の対応（Anthropic／OpenAI のSunset対応）
  10. 本番化判断のGate基準（PoCの何が満たされたら次フェーズに進むか）

特に重要なのは 2. と 3.。「生成AIを使います」だけで具体的なモデル名が出てこない見積書は、見積もり精度自体が怪しいです。Claude Sonnet と GPT-4o ではAPI単価が3倍以上違うので、月額試算が出せないのは設計が固まっていない証拠です。

「安い見積もり」が抱える典型リスク

複数社から相見積もりを取ったとき、極端に安い見積もりが1社含まれることがあります。生成AI開発で安すぎる見積もりに共通するパターンは次の通りです。

  • PoC兼プロトタイプとして提案されている: 評価設計が甘く、本番化で作り直しになる
  • GPT-3.5系・旧世代モデルを前提にしている: 精度が足りず再開発が必要
  • プロンプト1本書いて終わりの構造: 業務適用には精度・網羅性が不足
  • API利用料が見積もりに含まれていない: 後から「実費精算」で大きく追加請求

「生成AIだから安く・早く」というセールストークは、PoCレベルなら成立しますが、本番運用では破綻します。

まとめ: 発注者が事前に決めておくべき3つのこと

生成AI開発の見積もりを精度よく取るために、発注者側で事前に決めておくべきことは次の3つです。

  1. どのフェーズの費用なのか: PoC／プロトタイプ／本番化、どこまでの予算か
  2. データ範囲: どの社内データをLLMに参照させるか、機密区分はどうか
  3. 成功基準: 「使える」とは何が満たされた状態か（精度・速度・運用負荷）

この3つが言語化されていれば、見積もりの粒度はぐっと上がります。逆に言えば、これが曖昧なまま見積もり依頼すると、各社の見積もりが揃わずに比較できません。

弊社では、まず1〜2週間の要件定義スプリントでこの3つを発注者と握ってから本見積もりを出す形をとっています。「いくらかかるか」より先に「何を作れば成功か」を決める方が、結果として総コストが下がります。
