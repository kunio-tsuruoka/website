# 発注準備キット 改善プラン (2026-05)

## 背景

`/tools` 配下に発注検討者向けの3ツール（業務フロー可視化／ユーザーストーリー作成／スコープ管理）を公開済み。先行PR #20 で「発注準備キット」LPと業務順導線への再構成を完了。

ただし、ビジネスサイドが単体で完走できるレベルにはまだ達していない。本プランで段階的に改善する。

## 目的

- ツール完走率（中間CV）を計測可能にする
- 専門用語を減らしてビジネスサイドが詰まらない動線にする
- 出力した成果物が「次の打ち手」に自然につながる体験をつくる
- Beekleの世界観・専門性を伝える接点としてツールを最大化する

`.claude/rules/cro-strategy.md` の方針に従い、**ツールは完全公開のまま**改善する（メアド取得は別ルートのサービス資料DLで行う）。

## 実機監査結果 (Playwright, 2026-05-03)

`scripts/tool-ui-audit.mjs`（一時、後に削除）でdesktop/mobileのスクショ＋ボタン寸法＋用語頻度を計測。所見:

### 用語密度（1ページあたりの出現回数）

| ツール | As-Is | To-Be | BPMN | スイムレーン | FM/ファンクショナリティ | EARS | ユーザーストーリー |
|---|---|---|---|---|---|---|---|
| /tools | 1 | 1 | 0 | 0 | 1 | 1 | 1 |
| flow-mapper | **16** | **17** | 1 | 1 | 0 | 0 | 1 |
| story-builder | 0 | 0 | 0 | 0 | 0 | **3** | **6** |
| scope-manager | 0 | 0 | 0 | 0 | **5** | 0 | 4 |

**flow-mapperの As-Is/To-Be 30回超、scope-managerのFM/ファンクショナリティ5回**は致命的。タブ名・本文・ツール内ラベルすべてに散らばる。

### モバイル44px未満タップターゲット（実機計測）

flow-mapperで14個、story-builderで10個、scope-managerで7個、tools-indexで1個検出。特に深刻:

- flow-mapper: **削除「×」ボタンが16x16px** が3箇所、「閉じる ×」24x24px、「サンプルを読込」「接続モード」「全画面」「エクスポート」「リセット」が30x*px
- story-builder: **「ファイルを読み込む(.txt/.md/.docx)」が30x253px**（横長すぎ＋高さ不足）
- 関連リンク（/tools 戻り、姉妹ツールへのリンク）が全ページで17x*px

### 説明過多

flow-mapper のヒーロー後の説明テキスト = **3,480字**（tools-index 1,794字 / story-builder 1,868字 / scope-manager 1,973字）。スクロール多くツール本体到達まで時間がかかる。モバイルだと特に致命的。

### スクショから見えた構造的問題

#### tools-index ヒーロー
- バッジ「Tools」+ 大文字タイトル+本文6行で視覚的に地味
- モバイルではヒーローだけで1スクロール分占有、3ツールカードまで遠い

#### flow-mapper
- 業種別テンプレ選択モーダル（既存）はデスクトップ・モバイル両方で表示OK
- ただしモーダル下にうっすら見える「現状の業務フロー」キャンバスが消化不良な視覚
- 上部に「As-Is をスイムレーンで整理」「To-Be 改善ヒント」など専門用語見出しが連続

#### story-builder
- 入力直後に「正常系5件 異常系4件 ... 境界2件」のバッジが表示
- これは何か（生成結果の件数？入力ヒント？）文脈なしで意味不明
- AIサンプルが入った状態の textarea は触発的だが、`「正常系」「異常系」「境界」`は開発用語

#### scope-manager
- 初期画面が「ここにMarkdownを貼り付け、または「サンプルを試す」を押してください」だけ
- 「要求文を抽出」ボタンも何をするか不明瞭（青強調が弱い）
- FM 補足ボックスの「白川克 著」リンクは外部 amazon、自社の信頼資産から外に逃がす

## 現状と主要な課題（観察ベース）

### 課題1: 専門用語が散在 ★裏付けあり
flow-mapper で As-Is/To-Be の合計33回、scope-manager で FM/ファンクショナリティ 5回。ページタイトルはビジネス語化済みだが本文・ツール内UI・タブに残存。

### 課題2: モバイルタップターゲット不足 ★裏付けあり
3ツール合計31箇所が44px未満。コラム記事は4月末に修正したが**ツール側は未対応**。

### 課題3: 説明過多でツール本体まで遠い ★裏付けあり
flow-mapper 3,480字。スクロール多すぎ。モバイル離脱要因。

### 課題4: 中間CVが計測できていない
ツール起動 → 完走 → 出力 のどこで離脱しているか不明。GA4 にカスタムイベント未設定。

### 課題5: 初期状態の手がかり不足
特に scope-manager は textarea 1個と弱いボタン2個だけ。「サンプルを試す」の優先度を上げないと白紙離脱。

### 課題6: ツール間連携がない
3ツールが独立、出力を手動でコピペ。自然な工程を一気通貫で進められない。

### 課題7: 成果物の「次の打ち手」が薄い
Markdown／PDF出力後の使い道が見えない。最終アウトプット「RFPドラフト」未実装。

### 課題8: story-builder の謎バッジ
「正常系/異常系/境界 件数」が文脈なしで表示され混乱の原因。

## 改善方針

### 設計原則
- **ビジネス語優先**: 専門用語は残す場合もホバー注釈に格下げ
- **コピーから始める**: 白紙の代わりに編集可能なテンプレを最初に見せる
- **完走を計測する**: 主要アクション（保存・出力・サンプル読込）を全てイベント化
- **公開のまま改善**: メアド取得・会員登録は導入しない（CRO方針）
- **小さく出す**: 大規模再設計より、各フェーズで観察可能な改善を積む

### Phase別ロードマップ

| Phase | テーマ | 主たる修正対象 |
|---|---|---|
| **0** | 即修正 (UX/モバイル基本) | 44pxタップターゲット、初期画面の弱さ、謎バッジの説明追加 |
| **1** | 計測の土台 | GA4キーイベント化、ファネル可視化 |
| **2** | 用語ブラッシュアップ | As-Is/To-Be、FM/ファンクショナリティ、EARS の表記統一 |
| **3** | 説明圧縮＋スクロール短縮 | flow-mapper 3,480字を1,800字以下に、本体への到達短縮 |
| **4** | テンプレート投入（ツール内） | 経理締め・採用フロー・営業案件など業界別 |
| **5** | コラム末CTA連携 | カテゴリ別「関連ツールを試す」ブロック |
| **6** | ツール間データ連携 | flow-mapper → story-builder → scope-manager の連結 |
| **7** | RFPドラフト自動生成 | 3ツール出力を統合した1ドキュメント |
| **8** | ウィザード型オンボーディング | 質問3〜4問で最初のステップ自動生成 |
| **9** | シェアURL | localStorage圧縮→URL、読み取り専用ビュー |

時系列ではなく依存関係順。Phase 0/1/2 は並行可能、即着手。Phase 6 以降は計測データを見て要否を判断。

## Phase 0: 即修正できる UX 不備（観察ベース）

監査で見つかった具体的な不備を直す。仕様変更ではなく**修繕**レベル。

### やること

#### モバイル44px化
- flow-mapper の削除「×」ボタン 16x16 → **44x44** (`p-2.5` 等で領域確保)
- 「閉じる ×」24x24 → **44x44**
- ツールバーのボタン群 (サンプルを読込／接続モード／全画面／エクスポート／リセット) を mobile では `min-h-11` (44px) 適用
- story-builder の「ファイルを読み込む」ボタンを縦長レイアウトで44px
- 関連リンク（/tools 戻り、姉妹ツールへのリンク）に `py-2` 適用

#### 初期画面の手がかり追加
- scope-manager: 「サンプルを試す」ボタンを大きくして1stビューに目立たせる
- scope-manager: 入力エリアに「3行で例を見る」ようなインライン例を表示
- story-builder: 「正常系/異常系/境界 件数」バッジに `aria-label` ＋ 説明文 「(生成後の出力件数)」を併記。**初期表示は0件で隠す**
- flow-mapper のテンプレ選択モーダルが閉じた直後の「次の一手」の手がかり追加（"レーンを1つ追加→ステップを置く"の小さなチェックリスト）

#### 軽微な文言修正
- scope-manager の amazon 外部リンクは `rel="noopener nofollow"` を維持しつつ、近くにBeekleの関連コラムを併置（外への離脱を緩和）

### 成果物
- 各 feature の component 修正（モバイル class 追加、aria-label追加）
- 文言の小編集

### 依存
なし

### 想定インパクト
**中〜大**: モバイル離脱の即時改善。`/tmp/tool-ui-audit/` のスクショ観察ベース。

### 実装難易度
**小**: テキスト変更＋Tailwindクラス追加が中心、ロジック変更ほぼ無し

## Phase 1: 計測の土台 (GA4キーイベント化)

### やること
- 各ツールで以下を `gtag('event', ...)` で発火
  - `tool_start`: ツール本体マウント時、source（ツール名）を payload
  - `tool_save`: 自動保存／手動保存時
  - `tool_export`: Markdown／PNG／SVG／PDF 等の出力ボタン押下時、format を payload
  - `tool_load_sample`: サンプル読込ボタン押下時
  - `tool_complete`: 「完走」とみなす条件（例: ステップ3個以上 + 出力1回以上）に到達した時
- `setup-ga4-key-events.mjs` パターンでGA4 Admin APIから上記をキーイベント化
- Looker Studio に「ツール別ファネル」ダッシュボード作成

### 成果物
- `src/features/*/store.ts` に計測トリガ
- `scripts/setup-ga4-tool-events.mjs`
- ファネル可視化URL

### 依存
なし（即着手可能）

### 想定インパクト
**大**: 以降の改善判断の基礎データ。これがないと2以降の効果測定が不能。

### 実装難易度
**中**: 既存の `trackCtaClick` パターンを流用、各featureのstore actionに埋め込む

## Phase 3: 説明圧縮とスクロール短縮

### やること
flow-mapper のヒーロー後説明（3,480字）を1,800字以下に圧縮。重複を削除し、ツール本体に**スクロール1〜2回で到達**できるようにする。

具体的な圧縮対象:
- 「AI／DXは『業務可視化』から始める」セクション（既に冒頭サブタイトルで言及）→ 4行→1行に
- 「このツールでできること」3カードの本文をそれぞれ50字以内に
- 「こんな比較ができます」のサンプル数値表示は1テーブルに集約
- 「使い方（4ステップ）」を3ステップに（フェーズ・担当作成は手順1にまとめる）

scope-manager / story-builder も同様の圧縮（ただし元から短いので軽く）。

### 成果物
- 各 `src/pages/tools/*.astro` のヒーロー後セクション簡素化
- 削った説明は `/column` の関連記事へのリンクに置換（コラム流入の足しにもなる）

### 依存
Phase 2（用語と文言が同時に変わる）

### 想定インパクト
**中〜大**: モバイル直帰率の改善、ツール起動率の上昇

### 実装難易度
**小〜中**: コピーライティング判断が中心、エンジニアリング作業は少ない

## Phase 7: RFPドラフト自動生成

### やること
flow-mapper / story-builder / scope-manager の出力を統合して、発注時に開発会社へ渡せる **RFPドラフト** を自動生成する。

### 出力形式
- Markdown（編集可能）
- HTML/PDF プレビュー
- 章立て:
  1. プロジェクト概要（業務名・目的）
  2. 現状業務（flow-mapper の As-Is）
  3. 改善後の姿（flow-mapper の To-Be）
  4. やりたいこと一覧（story-builder のユーザーストーリー）
  5. 優先度判定（scope-manager のテーブル）
  6. 想定スコープ・除外スコープ
  7. （手入力欄）希望時期・予算感・連絡先

### 成果物
- 新規ツール `/tools/rfp-builder` または既存ツールの統合出力ボタン
- 章テンプレート群
- PDF出力（既存の svgToPng 同等のクライアント完結手法）

### 依存
Phase 6（連携できていることが前提）

### 想定インパクト
**大**: ツール群の最終アウトプット = Beekleの専門性の証明 = 強烈な差別化

### 実装難易度
**大**: PDF生成、章立てのテンプレ設計、既存3ツールの出力データ正規化

## Phase 8: ウィザード型オンボーディング

### やること
空キャンバスを最初に見せる代わりに、3〜4問の質問で最初のステップを自動生成する。

#### 質問例（flow-mapperの場合）
1. いまの業務でいちばん時間がかかっているのは何ですか？
2. 関わっている部署はどれですか？（複数選択）
3. その業務に関わる人数は？

→ 答えると最初のレーン2〜3本＋ステップ3〜5個が自動生成される。「普通モード」へのスキップボタン付き。

### 成果物
- 各 feature に Wizard コンポーネント
- 質問→初期データ生成ロジック

### 依存
Phase 4（テンプレが整ってからの方が質問を組みやすい）

### 想定インパクト
**中**: 初見ユーザーの完走率を底上げ。ただしテンプレで代替できる部分も多い

### 実装難易度
**中**: 上級者には邪魔にならない設計（dismiss 後は二度と出さない）

## Phase 9: シェアURL

### やること
作ったデータを **読み取り専用URL** として共有できるようにする。

### 実装方針
- `localStorage` のデータを LZString で圧縮 → URL クエリパラメータに乗せる
- 読み取り専用ビュー（編集UI非表示）を新設
- 共有ボタン押下時にクリップボードへコピー
- サーバーサイドストレージなし（公開ゲートで縛らない方針と整合）

### 成果物
- `src/features/*/share-url.ts` ユーティリティ
- 読み取り専用ビュー
- 共有ボタン UI

### 依存
Phase 1（共有URLからの来訪をUTMで計測したい）

### 想定インパクト
**中**: 上司レビュー・社内共有の用途を作る = 「使ったあとに人に見せる」体験

### 実装難易度
**小〜中**: 圧縮データの URL 長さ上限（2,083 chars on IE/古いブラウザ、Chromeは8,192）が懸念。サイズが超えたら fallback メッセージ

| 指標 | 計測元 |
|---|---|
| ツール起動数 | GA4 `tool_start` |
| 完走率 | `tool_complete` / `tool_start` |
| サンプル使用率 | `tool_load_sample` / `tool_start` |
| 出力数 | `tool_export` |
| コラム→ツール遷移率 | GA4 page_referrer + `tool_start` |
| 複数ツール完走率 | session 内の異なる `tool_complete` count |
| 共有URL経由の流入 | UTM parameter |
| ツールページからのお問い合わせ | source=tool-* の form_submit（スパム除外後） |

## やらないこと（YAGNI）

- 会員登録／メアド入力ゲート（CRO方針に反する）
- サーバーサイドストレージ（プライバシー価値の毀損）
- 多言語対応（日本語のみで十分、対象が日本市場の発注検討者）
- リアルタイム共同編集（オーバースペック）
- AIによる自動完成（やりすぎると「自分で考えてない」体験になる）

## 失敗のサイン

以下が観察されたら方針を見直す:

- Phase 1 実装後、`tool_start` が **月50を下回る** → そもそも認知不足、Phase 5（コラム連携）を優先
- Phase 4 投入後、**サンプル使用率が10%未満** → テンプレが業務とズレている、内容を見直し
- Phase 6/7 実装中に **依存ロジックが想定以上に停滞** → 単体ツール改善を優先、統合は次の改修サイクルへ

## 次の打ち手

すぐ着手するのは以下:
1. **Phase 0 開始**: 監査で見つかったUX不備の即修正（44pxタップ・初期画面・謎バッジ）

Phase 3 以降は Phase 1 のデータが十分に溜まってから優先順位を再評価する。
Phase 5 (コラム末CTA連携) は Phase 1 の計測が動き始めた時点で並行着手可能。
