# Column writing style (AIっぽさ排除)

MicroCMS の `columns` に入稿する記事本文・ドラフトを書く時のスタイル規約。

## AIっぽい言い回しを使わない

### 冗長な主語明示
日本語では主語を落とすのが自然。「自分が」「自身が」を機械的に付けて名詞を修飾するのは生成AI特有のクセが出やすい。

- ❌ 「**自分が思う**理想のスケジュール」
- ⭕ 「希望的観測のスケジュール」「願望ベースのスケジュール」「絵に描いた餅のスケジュール」
- ❌ 「**自分が考える**ベストプラクティス」
- ⭕ 「ベストプラクティス」「考えうるベストプラクティス」

検出: `grep -E '自[分身]が(思う|考える|感じる|抱く|描く)' draft.html`

### 過剰な強調定型句
論旨と関係なく強調だけしている定型句は AI 文の典型。

- ❌ 「〜に他なりません」
- ❌ 「〜と言っても過言ではありません」
- ❌ 「まさに〜と言える存在」「いわば〜のようなもの」(連発)
- ❌ 「実は〜なのです」(段落頭で連発)

具体名詞・動詞で言い切る方が信頼できる。比喩を入れる時は1記事1〜2回まで。

### その他のAI tells
- 同段落内で同じ語尾（〜です/〜ます）が4文以上続く → 体言止めや「〜ません」「〜でしょう」を混ぜる
- 箇条書きの各項目が「〜することが大事」「〜することが重要」で終わる → 動詞で言い切る
- 「結論」「ポイント」「まとめ」を1記事に3回以上書かない

## MicroCMS は HTML 入稿（Markdown 残骸を残さない）

`columns` は HTML フィールド。**`<pre>` 以外で `**bold**` `# heading` `[text](url)` `* list` などの Markdown 記法を残さない**。サニタイザは強調記号を解釈しないので、本文にそのまま `**` 記号が表示される事故になる（実例: `things-not-to-do-pm` で `**「自分が思う理想のスケジュール」**` が本文に出ていた、2026-05-03 修正）。

代わりに:
- 強調 → `<strong>...</strong>`
- 見出し → `<h2 id="...">...</h2>` `<h3 id="...">...</h3>`
- リンク → `<a href="/path">label</a>`
- 箇条書き → `<ul><li>...</li></ul>`

ドラフト末尾で必ず確認:
```bash
grep -nE '\*\*[^*]+\*\*|^#{1,4} |^\* |^- ' drafts/<slug>.html
# ヒットゼロが正常
```

## 入稿/PATCH 前のチェックリスト（コピペ用）

```bash
# AIっぽい冗長主語
grep -nE '自[分身]が(思う|考える|感じる|抱く|描く)' file
# 過剰な強調定型句
grep -nE 'に他なりません|と言っても過言ではありません' file
# Markdown 残骸
grep -nE '\*\*[^*]+\*\*|^#{1,4} |^\* |^- ' file
# どれもヒットゼロが目安
```

ヒットしたら言い換えてから `publish-drafts.mjs` または直接 PATCH する。

## 既存記事のスキャン

全記事を再スキャンするスニペット:
```js
// scripts/scan-column-style.mjs (作成するなら)
const arr = await fetchAllColumns();
const patterns = [
  /自[分身]が(思う|考える|感じる)/g,
  /\*\*[^*]{1,40}\*\*/g,
  /に他なりません/g,
  /と言っても過言ではありません/g,
];
for (const a of arr) for (const p of patterns) {
  const m = (a.content||'').match(p);
  if (m) console.log(a.id, p, m.slice(0,3));
}
```

## EARS記法は実務記事では日本語で書く（英語キーワード混在を避ける）

発注者（非エンジニア）向けの実務・テンプレート記事でEARS記法の例を書くときは、英語キーワードを日本語にベタ付けする形（`THE システム shall 〜する。` `WHEN 〜した時、`）を使わない。読みにくく、サイトの「発注者の側に立つ」方針に反する。

**日本語EARSの書き方**（パターン名は見出しで英語併記可）:
- ユビキタス（常時）: `システムは、〜すること。`
- イベント駆動: `〜したとき、システムは、〜すること。`
- 状態駆動: `〜である間、システムは、〜すること。`
- オプション: `〜の場合、システムは、〜すること。`
- 望まない事象: `〜した場合、システムは、〜すること。`

**例外**: `ears-requirements-syntax-guide`（EARS解説の専用記事）は、英語テンプレ `The 〈システム〉 shall 〈振る舞い〉.` の直下に日本語訳を併記したバイリンガル教材。正式キーワードを教える目的なので英語を残す。実務記事と教材記事で扱いを分ける。

**Why**: 2026-05-30、ユーザーが `requirements-definition-template` のEARS例（英語キーワード混在）を「日本語じゃないからわかりにくい」と指摘。全コラムをスキャンしたところ英語EARSは2記事のみ（template と guide）。template を日本語化、guide は教材性質のため据え置き。変換は `scripts/patch-ears-japanese.mjs`（行単位置換・dry-run既定・置換後に英語マーカー残存をassert）。

# コラム表記規約: スラッシュの全角/半角

- **英語の技術用語ペア**（As-Is/To-Be 等）: **半角スラッシュ・スペースなし**（`As-Is/To-Be`）
- **和語・日本語並列**（AI／DX、作る／後回し／作らない 等）: **全角スラッシュ `／`**

2026-05-31 のWebコラム監修対応でユーザー承認。監修指摘 0039（タイトルの As-Is/To-Be を全角に、という指摘）は、この規約では As-Is/To-Be は半角が正のためスキップ。0052/0037 はこの規約で統一済み。新規コラム執筆・既存修正時はこの規約に従う。

# 生成AIの「活用課題 X%」統計は捏造せず帝国データバンク調査を出典付きで引く

生成AI系コラムの導入で「生成AI活用の課題調査で、X%の企業が『○○』を挙げています」と**出典なしの小数点%を書かない**（marketing.md「丸い%は使わない」/ ハルシネーション防止と同根）。

検証可能な一次ソース: **帝国データバンク「生成AIに関する企業の動向調査」（2026年3月、有効回答1万312社）** https://www.tdb.co.jp/report/economic/20260514-genai/
- 活用している企業 34.5%（非常に4.4＋やや30.2）
- 課題: 情報の正確性50.4% / 専門人材・ノウハウ不足41.3% / 活用すべき業務の範囲40.0% / 情報漏洩リスク33.5% / ルール整備25.5%

注意: この調査の課題カテゴリは5つ程度。「経営者の理解」「既存システム制約」「ROI測定」「組織体制」「データ基盤」等のニッチテーマに対応する個別%は実在しないので、**無理に数字を当てない**。直接対応が無いテーマは活用率34.5%や近いカテゴリで開き、テーマは定性的に繋ぐ。引用は `<a href>` で出典明記。

事例 2026-07-01: 生成AI系9本の冒頭が出典不明の統計（41.5/32.5/26.0/25.5/18.5/12.0/9.0/7.0%）だったため、TDB実数値+出典リンクに全置換（PATCH済み・検証済み）。

# 「経験則の一般化」は会社の見解で済ませず、帰属＋前提を付ける

「ほとんどのケースで○○で十分」「大半は△△は要らない」のような**反証可能な経験的一般化**を、「会社の見解だから事実確認不要」と扱わない。これは隠れた事実主張（分布の主張＋能力の主張）を含み、無防備に断定すると「根拠なき断定」になり、捏造統計（[[出典なき統計→TDB差し替え]]）と同じ穴に落ちる。

正しい書き方（会社の断定スタンスは維持しつつ）:
- **帰属**: 「私たちが手がける案件では」「私たちの経験では」と、測った統計ではなくBeekleの観測だと分かる形にする。
- **前提（バウンド）**: 能力主張には条件を添える（例「想定内の負荷ならVPSで十分捌ける」）。無条件の能力断定にしない。
- 公開統計が存在しない主張に「○%」の数字を作らない。

事例 2026-07-01: 生成AIインフラ記事で「ほとんどVPSで十分」を当初フラットに断定→ユーザー指摘で「経験では」「想定内の負荷なら」を付与。会社の強い立場（安い・速い・VPSで十分。[[feedback_kintone_vs_genai_stance]]と同系）は崩さず、誠実さと両立させる。

# ナレッジグラフ(KG)の公開事例（一次ソース検証済み・RTB用）

KG/GraphRAG系コラムで「世の中の事例」として使える、一次ソースで裏取り済みの数字（2026-07-01検証）。コラムに載せる際は必ず出典リンク付きで引用する（出典なし数字は禁止＝今セッションで9記事を是正した教訓と同じ）。

- **Meta**：データ基盤の暗黙知を50+AIエージェントで構造化。AIコンテキストカバレッジ 約5%→100%、AIツール呼び出し40%削減。出典: https://engineering.fb.com/2026/04/06/developer-tools/how-meta-used-ai-to-map-tribal-knowledge-in-large-scale-data-pipelines/ （Meta Engineering, 2026-04-06）
- **LinkedIn**：過去サポートチケットをKG化したRAGで、問い合わせ解決時間（中央値）を28.6%短縮。出典: https://arxiv.org/abs/2404.17723 （LinkedIn著者の論文）。※「品質改善28.6%」は不正確、正しくは「解決時間28.6%短縮」。
- **NTTデータ**：ニュース記事からLLMで企業間関係を自動抽出、精度約73%。投資判断・競合分析・サプライチェーン可視化に活用。出典: https://www.nttdata.com/jp/ja/trends/data-insight/2024/1108/

掲載先: `/column/knowledge-graph-rag-business` の「世の中の事例：数字で見るナレッジグラフの効果」節。スライド等の二次画像は著作物なので埋め込まず、事実を一次ソースから引用する。

# コラム本文の <strong> は使いすぎない（強調の氾濫を避ける）

コラム本文で `<strong>` を多用すると視線が散り、強調が機能しなくなる（ワーキングメモリ負荷大）。1段落に何度も太字、ラベル（質問/ふつうのAI 等）と本文強調を両方太字、は避ける。本当に効かせたい語だけに絞る。

レンダリング側（2026-07-01, PR #61）: `src/pages/column/[...slug].astro` の `.markdown-content strong` から薄紫マーカー（linear-gradientハイライト）を撤去し、**太字＋濃色(text-gray-900)のみ**に変更済み。以前は全 strong にハイライトが付き「全部光って見える」状態だった。FAQ質問用の別ルールは残置。

検証手法: dev起動→Playwright(chromium, 1280/390px)で実記事をスクショし、自分(vision)で before/after を確認するのが有効。スクリプトは node_modules 解決のためリポジトリ直下に置いて実行する（scratchプレースだと playwright が解決できない）。

# 日本人読者向けコラムに英語記事の「参考にしています」引用を入れない

日本語コラムの読者は日本人。海外ブログ等を「〜の考え方を参考にしています」と本文に英語URLで引用しない（読者が辿れず信頼にも繋がらない）。着想元が海外記事でも、本文には出典リンクを出さず、内容だけ日本語で咀嚼して書く。

- 事例 2026-07-05: ai-agent-capability-upgrade で PuppyGraph「Knowledge Graph Agents」への参考リンクをユーザー指摘で削除。
- 例外（グレー）: 数値の根拠として引く統計の出典（Meta Engineering / arXiv 等の英語一次ソース）は、捏造防止のため出典明記が必要で別扱い。ただし可能なら日本語の一次ソース（例: NTTデータ DATA INSIGHT）を優先する。attributional な「参考」リンクと、evidence（数値の裏取り）の出典は区別する。
- 関連: [[column-writing-style]]（AIっぽさ排除／出典なき統計はTDB等で裏取り）

# 技術用語は直訳・造語より定訳/簡潔な日本語を使う

海外記事の概念を訳すとき、直訳っぽい造語を本文に置かない。既に流通している定訳か、簡潔な日本語表現を選ぶ。

- 事例 2026-07-05 (ai-agent-capability-upgrade): "persistent memory" を「続く記憶」（直訳っぽい）→「覚え続ける」（まだ不自然）と迷走し、最終的にユーザー指示で**「持続記憶」**（定訳）に着地。初回訳語で `direct-translation臭` が出たら、造語を練り直すより定訳を調べて充てる。
- 一度括弧で軽く言い換えを添えるのは可（例:「持続記憶（やり取りをまたいで消えない記憶）」）。
- 関連: [[column-writing-style]] のAIっぽさ排除（直訳調・不自然な語尾）。
